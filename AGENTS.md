# AGENTS.md

## 1. 项目目标
本项目用于构建一个确定性的混合架构自主网络智能体，并逐步演进为 **多数据源岗位聚合与分析平台（Multi-Source Job Orchestrator / Dashboard）**。

默认采用 **deterministic heuristic-first** 策略：
- 优先使用预定义规则、状态机、选择器、解析器、schema、字段映射和可验证逻辑完成任务
- 仅在启发式逻辑无法可靠覆盖、页面交互未知、或解析结果不确定时，才升级到 VLM/LLM fallback
- fallback 仅用于兜底，不可反客为主替代主流程

当前产品定位不是官网，也不是 AI 概念稿；它首先是一个 **高信息密度的数据产品 / SaaS 风格内部工具 / 招聘聚合 dashboard**。

---

## 2. 规则优先级（从高到低）
当规则冲突时，按以下优先级执行：

1. **安全与密钥规则**
2. **用户明确要求**
3. **本 AGENTS.md**
4. **项目内更具体的子目录 AGENTS.md / 局部规范**
5. **`.agents/skills/` 下的技能规则**
6. **默认工具行为**

---

## 3. 全局行为规范

### 3.1 语言
- 与用户的所有交流、总结、计划、说明必须使用 **简体中文**
- 代码、命令、日志、协议字段、报错信息可保留原文
- 内部思考语言不限

### 3.2 搜索与验证
- 需要联网搜索时，**优先使用 PPLX / Perplexity**
- 若 PPLX 不可用、结果不充分、或任务明显更适合本地代码/本地文档验证，则允许降级到其他可靠来源
- **代码事实、运行结果、测试结果优先级高于搜索结果**
- **强制项目进度同步 (Mandatory Progress Sync)**：
  - 每次对话开始前，代理**必须**主动读取根目录下的 `PROJECT_PROGRESS.md`
  - 每次完成一项功能闭环、重大重构、目录迁移、接口调整、UI 大改或规则更新后，**必须**同步更新 `PROJECT_PROGRESS.md`
  - 切换代理前，必须确保项目进度文档足够让下一个代理无缝接手

### 3.3 清理
- 创建的临时脚本、测试日志、快照、调试产物必须在任务结束前清理
- 除非用户明确要求保留，否则不要把临时文件留在项目根目录
- 推荐放在 `tmp/`、`.tmp/`、`artifacts/` 等约定目录中并在结束时删除

### 3.4 沟通
- 不要只汇报“做了什么”，要说明：
  - 为什么这样做
  - 当前结论
  - 风险或未完成项
- 若存在不确定性，必须显式说明，不可假装确定
- 遇到结构性问题，不要只做局部缝补；应先判断是否需要目录重构、边界重划或职责拆分

### 3.5 产品与界面定位
- 本项目的前端定位是 **高信息密度的数据产品 / 招聘聚合 dashboard / 内部工具风格 UI**
- **不是**营销官网、产品展示页、AI showcase、概念海报或未来感实验页面
- 所有前端实现必须优先服务：
  - 信息扫描效率
  - source 间比较效率
  - 筛选与排序效率
  - 列表可读性
  - 统计与活动趋势的可理解性
- 禁止为了“高级感”引入与任务无关的大面积装饰性设计，例如：
  - 夸张渐变
  - 玻璃拟态 / 毛玻璃泛滥
  - 霓虹 / 发光边框
  - 厚重阴影
  - 超大圆角
  - 过度留白
  - 把表格或分析页做成展示型卡片墙
- 默认视觉方向应接近成熟 SaaS / admin / internal tool，而非 AI demo 风格

### 3.6 Agent Skills 约定
- 跨模型兼容的项目级技能统一放在：**`.agents/skills/`**
- 若某个模型同时支持其他路径（如 `.claude/skills/`），也**不要**在本项目中额外复制一份，避免多处漂移
- 技能规则只作为增强，不得替代本 AGENTS.md 的全局规则
- 新增或修改技能后，应在总结中说明：
  - 路径
  - 作用
  - 自动触发场景
  - 是否影响已有工作流

## 4. 默认执行策略

### 4.1 复杂任务默认流程（Mandatory）
以下任务必须走完整 workflow：
- 新功能开发
- 多文件修改
- 外部系统 / MCP / 网络集成
- 架构调整
- 目录重构
- 前后端边界重划
- 涉及测试、发布、鉴权、安全的改动
- 用户要求“先设计再实施”的任务

默认流程如下：
1. `using-superpowers` / `brainstorming`
2. `using-git-worktrees`（适用时）
3. `writing-plans`
4. `subagent-driven-development` / `executing-plans`
5. `test-driven-development`
6. `requesting-code-review`
7. `finishing-a-development-branch`

### 4.2 快速通道（Fast Path）
以下低风险任务可不走完整流程，但必须在回复中简述理由：
- 纯分析、纯解释、纯审阅
- 单文件小改动
- 文档修正
- 注释、命名、格式化等低风险修改
- 明显无需设计分解的一步式任务

即使走 Fast Path，也仍需遵守：
- 安全规则
- 清理规则
- 必要验证
- 最终说明改动和风险

### 4.3 前端改动默认策略
凡是涉及 dashboard、table、tabs、filters、stats、activity chart、schema 可视化、详情页的任务，默认遵循以下顺序：

1. 先读取 `PROJECT_PROGRESS.md`，确认当前阶段与关键模块
2. 先判断是“视觉问题”还是“结构问题”
3. 优先做最小必要改动，不擅自改动 API contract、路由、数据库语义或字段含义
4. 优先复用现有组件、样式 token、布局模式与图表 / 表格基础设施
5. 优先提高信息密度、可扫描性和一致性，而不是增加装饰效果
6. 如需重构，必须说明：
   - 哪些是视觉层改动
   - 哪些是结构层改动
   - 是否影响现有行为

### 4.4 目录与职责边界默认目标
当项目进入“前后端分离 / 目录重构 / 模块化”阶段时，优先靠拢以下职责边界：

- `apps/web`：前端应用，仅包含 UI、页面、状态、视图逻辑、前端 API 调用
- `apps/api`：服务端 API、路由、聚合查询、触发入口、服务编排
- `packages/shared`：共享类型、schema、contracts、常量、通用 helpers
- `packages/ui`（如确有必要）：可复用 UI primitives，不为拆分而拆分
- `services/ingest` 或 `apps/api/src/ingest`：各 source adapter、抓取、清洗、去重、路由逻辑
- `db/`：migration、schema、seed、DB 文档
- `scripts/`：一次性脚本、维护脚本、迁移辅助脚本

禁止：
- 前端直接依赖抓取实现细节
- 后端目录与页面组件相互缠绕
- 把 schema / adapter / crawler / route / UI 混放在同一层级
- 为了“看起来高级”而进行无收益的过度拆分

## 5. Superpowers 工作流要求

### 5.1 Brainstorming
在编写代码前，先明确：
- 用户目标
- 成功标准
- 已知约束
- 未知项
- 风险点
- 是否适合 heuristic-first
- 是否需要目录重构或职责重划

禁止在需求和边界未明确时直接进入大规模编码。

### 5.2 Planning
计划必须是可执行的，至少包含：
- 步骤拆解
- 每步预期结果
- 验证方式
- 失败时回退策略
- 若涉及目录迁移，必须附带迁移后职责说明

### 5.3 Execution
- 优先实现最小可验证版本
- 优先改最少文件
- 优先使用可解释、可重复的方案
- 不要为了“智能”而牺牲确定性
- 目录重构时，先稳定边界，再迁移代码，再修脚本与导入路径

### 5.4 TDD
所有新增行为默认遵循：
- 红灯：先写失败测试或构造失败验证
- 绿灯：最小实现让其通过
- 重构：清理结构但不改变行为

如果任务不适合严格 TDD，必须至少做：
- 变更前基线验证
- 变更后回归验证

### 5.5 Review / Finish
结束前必须检查：
- 是否符合原计划
- 是否引入多余复杂度
- 是否遗留临时文件
- 是否更新必要文档
- 是否更新 `PROJECT_PROGRESS.md`
- 是否说明剩余风险
- 是否新增或修改了 `.agents/skills/`
- 是否出现前端风格跑偏或 AI 味回潮

## 6. 本地可用 MCP / 工具
如需与外部工具通信，优先使用已配置的本地 MCP：

1. **Zap MCP**: `http://localhost:7456`
2. **Kali MCP**: `http://localhost:8000/sse`
3. **Gemini API**: 通过环境变量 `GEMINI_API_KEY` 配置

使用原则：
- 能本地完成，先本地
- 能确定性完成，不先上 VLM
- 调用外部工具前先说明用途与预期产出
- 外部结果必须经过本地逻辑或测试验证，不可盲信

## 7. 安全与配置规则
- 严禁提交包含密钥、token、cookie、凭证的 `.env` 或等效文件
- 严禁把敏感值写入测试快照、日志、错误输出或提交信息
- 未经明确授权，不执行破坏性操作（删除生产数据、强制覆盖、批量外发、对外扫描）
- 涉及安全测试时，默认仅在授权边界内进行
- 目录重构或脚本迁移时，不得意外改变生产配置读取路径

---

## 8. Heuristic-first 强制准则
所有自动化任务必须优先尝试：
- DOM / selector / schema / regex / parser / rules engine / state machine
- 明确的字段映射与表单策略
- 可重放的动作序列
- 可审计的错误分类

只有在以下情况才允许升级到 VLM：
- 页面结构动态变化且难以稳定定位
- OCR / 视觉识别确有必要
- 启发式结果置信度不足
- 已有规则连续两次失败且无法在当前上下文快速修复

### 8.1 Dashboard 领域强制准则
针对本项目当前 v2.1 架构，以下领域约束默认成立：

- `src_linkedin` / `src_devitjobs` / `src_reed` / `src_jooble` / `src_hn_hiring` / `src_remoteok` 为独立 source schema
- `public.jobs_all` 为统一查询视图
- `public.content_index` 为跨平台关联索引
- 前端展示与交互不得误导这些关系，不得将其简化成错误的数据模型

以下模块属于核心一等公民，不得因 UI 美化而弱化：
- source tabs
- time range filter
- sort controls
- stats summary
- hourly activity
- duplicates review
- source metadata
- trigger / refresh 状态反馈

默认页面优先级：
1. 先保证数据浏览与筛选效率
2. 再保证统计与趋势表达
3. 最后才做视觉 polish

## 9. 失败升级机制（Mandatory PUA Trigger）
当出现以下任一情况时，必须触发 `pua` 或等效高强度复盘流程：
- 同一目标连续失败 >= 2 次
- 代理提出“让用户手动操作”之前，尚未穷举主要自动方案
- 代理准备放弃
- 输出明显缺乏推进性，只重复现状
- 问题被错误归因为“无解”，但实际上仍有可尝试路径

触发后必须输出：
1. 失败原因分类
2. 已尝试方案
3. 未尝试但值得尝试的方案
4. 最小下一步实验
5. 是否需要降级、绕路或切换工具

---

## 10. 完成定义（Definition of Done）
任务完成前，至少满足以下条件：
- 用户目标已被直接推进，而非只做局部分析
- 相关代码 / 配置 / 文档已同步
- 必要测试或验证已完成
- 临时文件已清理
- 风险、限制、未完成项已明确说明
- 没有引入未经解释的复杂度
- 若任务涉及目录重构，必须说明新的职责边界
- 若任务涉及前端改动，必须说明为什么现在更像成熟产品而非 AI demo

---

## 11. 默认输出模板
执行开发任务时，优先使用如下结构汇报：
1. 目标
2. 约束 / 假设
3. 计划
4. 执行结果
5. 验证结果
6. 风险与后续建议

### 11.1 前端任务输出补充模板
涉及前端页面、组件、dashboard 改动时，优先使用如下补充结构汇报：

1. 当前页面问题
   - AI 味 / 演示感体现在哪
   - 信息密度问题
   - 结构层级问题
2. 改动策略
   - 保留哪些逻辑
   - 调整哪些视觉层
   - 是否有结构调整
3. 执行结果
4. 验证结果
5. 风险与未完成项
