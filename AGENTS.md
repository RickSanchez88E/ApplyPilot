# AGENTS.md

## 1. 角色与目标
- 本仓库是一个多源职位抓取、调度、分析与表单入口解析项目，不是营销官网。
- 默认产品形态是高信息密度的 dashboard / internal tool，而不是 AI demo。
- 默认技术方向是 heuristic-first：优先规则、状态机、明确接口、可验证链路；模型兜底只能是 fallback。

## 2. Mandatory Read Order
每次开始任务前，必须按以下顺序读取上下文，不允许跳过：
1. 根目录 `AGENTS.md`
2. 根目录 `PROJECT_PROGRESS.md`
3. 与本次任务相关的局部 `AGENTS.md`
4. `.agents/skills/*/SKILL.md`
5. 当前任务输入与相关代码

如果存在 `scripts/agent-harness.ts`，优先运行它生成上下文摘要，再进入代码探索。

## 3. Harness First
- 默认入口：`npx tsx scripts/agent-harness.ts`
- 目标目录任务可使用：`npx tsx scripts/agent-harness.ts --target frontend --task "..." --format markdown`
- harness 负责统一读取规则、长期记忆、skills 和局部规范，避免漏读。
- harness 的输出是上下文摘要，不替代代码阅读；进入实现前仍要阅读相关文件。

## 4. Progress Sync Contract
- 完成功能闭环、架构调整、目录迁移、调度变化、浏览器策略变化、Docker 运行方式变化后，必须更新 `PROJECT_PROGRESS.md`。
- 更新原则：写“下一位 agent 必须知道的长期事实”，不写聊天流水账。
- 若本次改动没有落到 `PROJECT_PROGRESS.md`，默认视为交接不完整。

## 5. Memory Discipline
- `PROJECT_PROGRESS.md` 是长期记忆入口，不是日报。
- 必须优先记录：
  - 当前阶段与运行模式
  - 已锁定决策
  - 关键模块现状
  - 未完成事项
  - 当前风险
  - 最近验证基线
- 过期、失效、与下一次决策无关的历史细节应删除或压缩。

## 6. Decision Lock
- 已写入 `PROJECT_PROGRESS.md` 的锁定架构决策，不得擅自推翻。
- 若确需变更，必须在实现前说明变更原因，并在实现后同步更新 `PROJECT_PROGRESS.md`。
- 常见锁定事项包括：主技术栈、数据库真相源、队列模型、浏览器策略、调度方式、表单解析边界。

## 7. 全局行为规范

### 7.1 语言
- 与用户的所有说明、计划、总结必须使用简体中文。
- 代码、命令、日志、报错、字段名可保留原文。

### 7.2 搜索与验证
- 代码事实、运行结果、测试结果的优先级高于搜索结果。
- 需要联网搜索时，优先使用已配置工具；能本地验证就不靠外部搜索。
- 修改前先读上下文，修改后必须做最小必要验证。

### 7.3 清理
- 调试脚本、临时日志、快照等临时产物默认放 `tmp/` 或 `artifacts/`，任务结束前清理。
- 不要把一次性调试文件留在仓库根目录。

### 7.4 沟通
- 不只汇报“做了什么”，还要说明“为什么这样做、当前结论、剩余风险”。
- 有不确定性必须明确说出，不允许假装确定。

## 8. 前端默认约束
- 前端定位是成熟 SaaS / admin / internal tool。
- 不要做 AI demo 风格，不要堆夸张渐变、毛玻璃、霓虹、厚阴影、大圆角。
- 优先提高信息密度、可扫描性、排序与筛选效率。
- 平台页、总览页、统计区、表格区必须属于同一设计语言。

## 9. 执行策略
- 复杂任务默认：先读规则与记忆，再读相关代码，再做计划，再实现，再验证，再更新记忆。
- 简单任务也不能跳过 `AGENTS.md` 与 `PROJECT_PROGRESS.md`。
- 不为“显得高级”引入无收益复杂度。

## 10. Output Contract
最终汇报必须显式区分：
1. 已完成项
2. 未完成项
3. 验证结果
4. 剩余风险

如果是纯分析/审计，也要明确指出：
- 哪些结论已被代码或运行结果证实
- 哪些只是推断

## 11. Skills 约定
- 项目级 skills 统一放在 `.agents/skills/`
- 不在仓库内复制多个技能目录
- 使用 skill 后，应在总结中说明触发的 skill 或等价规则

## 12. 安全与配置
- 不提交 `.env`、token、cookie、真实凭证。
- 未经明确授权，不执行破坏性删除、批量对外发送或越权扫描。
- 涉及浏览器 profile、代理、登录态时，优先保守处理，不做“万能 token 注入”。

## Design Context

### Users
- 主要用户是运营/采集同学，使用场景是持续监控抓取链路、定位异常来源、评估可用职位质量。
- 首要目标不是展示，而是提升排障和决策效率。

### Brand Personality
- 调性：克制 / 专业 / 可靠。
- 需要传达高信任和低噪音，不追求消费级“活泼感”。

### Aesthetic Direction
- 默认 Light only。
- 视觉语言对齐成熟 SaaS / admin / internal tool。
- 明确避免消费级 App 风格和展示型装饰设计。

### Design Principles
1. 数据语义优先：UI 指标口径必须与 PostgreSQL 真相源一致。
2. 信息密度优先：提升扫描效率，减少无效留白与装饰。
3. 可访问性前置：按接近 AAA 要求约束文本、对比度与交互。
4. 反馈即状态：动画仅用于状态变化与任务反馈，不做炫技。
5. 一致性优先：总览、平台页、统计区、表格区使用统一设计语言。
